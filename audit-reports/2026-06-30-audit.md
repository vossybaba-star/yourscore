## High

### H-prior — anon key can read quiz `answer` columns (fraud path on the £100 board)
**Where:** `questions` (and `quiz_packs`) public SELECT policy `USING(true)` — [supabase/schema.sql:395](supabase/schema.sql).
**Re-confirmed live this audit (anon key only):** `GET /rest/v1/questions?select=id,answer` returns plaintext answers (`"A"`, `"B"`, …) for every question. A scripted client can pre-fetch answers for any quiz-gated content, defeating ranked-band / Mastermind integrity. Open since Jun 12.
**Note for this diff:** the new h2h challenge flow is *not* made worse by this — `/api/h2h/play` re-grades server-side from `quiz_packs.questions` via the service role, so a client knowing the answers still cannot forge an opponent score. The exposure remains the prize-board ranked content.
**Fix:** strip `answer` (and decider columns) from anon-readable grants — column-level `REVOKE`, or serve questions through a server route that omits the answer and grades server-side only.

### H1 — `profiles.timezone` / `country` location PII is anon-readable (carryover, still LIVE)
**Where:** [supabase/migrations/57_profile_timezone.sql](supabase/migrations/57_profile_timezone.sql) adds `timezone` + `country` with no column-level REVOKE, inheriting the permissive public `profiles` SELECT policy.
**Re-confirmed live this audit:** `select=username,timezone,country&timezone=not.is.null` returns named coarse-location rows with the anon key (e.g. `{"username":"dreamteam","timezone":"Europe/London","country":"GB"}`). Named, per-user, enumerable in bulk; grows every time `<TimezoneSync/>` runs. These columns are only ever read server-side (the hourly send-window cron, service role).
**Fix:** `REVOKE timezone, country, active_hour_utc ON public.profiles FROM anon, authenticated;` captured in a one-line migration (so the grant state lives in the repo), or move the notification-targeting columns to a service-role-only sidecar table.

## Medium

### M-new — `h2h_challenges` public `USING(true)` SELECT now leaks the targeted-challenge social graph (widened by migration 55)
**Where:** legacy policy `"Anyone can read h2h challenges" … to public using (true)` ([supabase/schema.sql:393](supabase/schema.sql)); migration 55 adds `invited_user_id` (and the inbox uses `opponent_id`) to the same table — [supabase/migrations/55_h2h_async.sql](supabase/migrations/55_h2h_async.sql).
**Confirmed live (anon key):** `GET /rest/v1/h2h_challenges?select=challenger_id,invited_user_id,opponent_id,challenger_name,challenger_score` returns every row. Today `invited_user_id` is still null on the legacy link-based rows, but Step 4 (challenge-a-friend entry points, commit `7ed1df4`) populates it — at which point the anon key can enumerate a **named, directional "who challenged whom" graph** (challenger → invited friend → opponent), joinable to public usernames, plus everyone's scores.
**Why it matters:** the table was designed for *open link-based* reads (anyone with a share URL sees one challenge). It is now also an *inbox* keyed to specific users, but the read policy never tightened — so a coarse social graph + score history is bulk-harvestable with the public key. Lower severity than H1 (no email/phone/location in this table; content is quiz scores), but it's a new privacy regression introduced by this week's work.
**Fix:** scope the SELECT policy to participants and open challenges, e.g.
`USING (invited_user_id IS NULL OR auth.uid() IN (challenger_id, invited_user_id, opponent_id))`.
The public/anon share-link accept page should fetch a single challenge via a server route (service role, by id) rather than relying on a blanket anon read.

### M1 (carryover) — daily push log-before-send can silently drop a whole cohort
[src/lib/notify.ts:54-61](src/lib/notify.ts) still inserts `notification_log` **before** awaiting `send-push`; per-day dedup + single-UTC-hour targeting means any transient `send-push` failure permanently loses that cohort's daily ping with no retry. Acknowledged in the code comment. Unchanged.
**Fix:** log after a confirmed 2xx (accept rare double-send), or delete the just-inserted rows on non-2xx; at minimum surface a `delivered` count.

## Low

### L-new — `h2h_challenges` INSERT policy still permits direct anon-client inserts → unbounded inbox injection (latent)
**Where:** policy `"Authenticated users can create challenges" … with check (challenger_id = auth.uid())` ([supabase/schema.sql:394](supabase/schema.sql)). No app code uses it anymore (all creation goes through `/api/h2h/create`, which is rate-limited 20/60s), but the policy lets any authed user `INSERT` a challenge straight via the anon client — setting `invited_user_id` to **any** victim and an arbitrary `challenger_score`, bypassing the API rate limit, dropping rows into a stranger's Your-Turns inbox. Impact is low today (names + scores; no email is wired yet — `/api/h2h/create` still has the Step 6 email send as a `TODO`), but it becomes a spam/notification-abuse vector the moment the "X challenged you" email lands.
**Fix:** since creation is now exclusively server-side, drop the client INSERT policy (writes already use the service role). Also add a friend/UUID-format check on `invitedUserId` in [create/route.ts](src/app/api/h2h/create/route.ts) before the email step ships — currently any well-formed UUID is accepted as a target with no relationship check.

### L-cosmetic — challenger's own score in `/api/h2h/create` is client-supplied
[src/app/api/h2h/create/route.ts](src/app/api/h2h/create/route.ts) takes `score`/`correct` from the client (validated only `≥0, finite`); only the *opponent's* score is server-graded. The route comment acknowledges this ("same trust model as before"). Fine for friend-vs-friend bragging, but means the challenger half of any result card is not authoritative. Leave as-is unless h2h ever feeds the ranked board.

### L1 (carryover) — **FIXED.** Daily quiz blast now on Broadcasts, not the transactional quota
[scripts/send-wc-quiz-daily.mjs:24,56](scripts/send-wc-quiz-daily.mjs) now calls `syncAndBroadcast` ([scripts/lib/broadcast.mjs](scripts/lib/broadcast.mjs)) on `RESEND_CAMPAIGNS_API_KEY` (commit `9b09a7d`), so the ~3,275/day send no longer shares the over-quota transactional pool with password resets / magic links. Verified in the diff. Closing this item.

Still absent locally; the webhook is fail-closed (503 without it), so if it's also unset in prod all bounce/complaint suppression silently stops. Confirm it's set in Vercel.
