**Branch/HEAD:** `daily-quiz-launch-rebuild` @ `32806c4` (Add user segmentation system). Audit covers the 4 commits since the Jun-21 base `f7c3446`: the Resend-Broadcasts switch, the daily-quiz launch rebuild, the WC quiz sub-category, and the user-segmentation system (migration 55).

## Executive summary
**No new Critical or High *security vulnerability* in deployed app code this cycle.** The three app-facing changes are sound: the webhook hardening is a real improvement (now fail-closed + suppress-only + syncs broadcast unsubscribes), the new World Cup quiz tab is pure UI, and migration 55's segment functions are correctly `security definer`, `search_path`-pinned, and locked to `service_role` (the PUBLIC-default grant gotcha is handled). **The one finding that rises to High is a provider-limit/deliverability problem, not a vuln:** the **daily** all-users quiz email still ships through the **transactional** Resend API — the quota that memory shows already *over* (57,676/50,000) — even though the campaign emails were moved to Broadcasts this cycle. A secondary Medium: the Broadcasts contact-upsert forces `unsubscribed:false`, which can resurrect users who unsubscribed via Resend's own link unless the (still-unconfirmed-in-prod) `contact.unsubscribed` webhook has captured them. The long-standing integrity items — **C1** (WC answer bank in client JS) and **H-prior** (anon `SELECT` on quiz answers) — remain open.

## High

### H1 (NEW — provider limit / deliverability) — Daily quiz email blasts ALL users via the over-quota *transactional* API
**Files:** `scripts/send-wc-quiz-daily.mjs:58,167,188` (uses `new Resend(RESEND_API_KEY)` + `resend.batch.send`, tagged `category:campaign`), invoked daily by `scripts/launch-daily.mjs:131` (`send-wc-quiz-daily.mjs … --send`).
**What's wrong:** This cycle moved the **mastermind launch + lapsed** campaigns to Resend **Broadcasts** (`send-wc-mastermind-*.mjs` → `RESEND_CAMPAIGNS_API_KEY`/`syncAndBroadcast`) — the correct relief path. But the **highest-frequency send**, the daily WC-quiz email to *all* sendable users (~3,275/day), was **not** migrated. It still fans out one transactional message per user through `RESEND_API_KEY`.
**Why it matters:** ~3,275/day × 30 ≈ **~98k/month against a 50k transactional cap** that memory already records as *over quota* (57,676/50,000). The transactional quota is shared with genuinely transactional mail — **password resets and magic-link sign-ins**. Over-quota means overage billing and/or throttling, and a throttle on this pool degrades auth email (users can't reset/sign in) while also silently dropping daily-quiz recipients. The whole point of the Broadcasts switch (commit `4890438`) was to take campaign volume *off* this quota; leaving the daily send on it defeats that for the biggest contributor.
**Suggested fix:** Route the daily quiz email through Broadcasts like the mastermind sends — `broadcastToSegment(RESEND_CAMPAIGNS_API_KEY, …, { segment: "all_sendable" /* or "engaged" */, … })`. That also gets per-recipient `{{ subscriber.unsubscribeUrl }}` and one-list management for free. Keep `RESEND_API_KEY` strictly for auth/transactional. If staying on transactional short-term, at minimum target an **engaged** segment, not all users, to stay under cap.

## Medium

### M1 (NEW — compliance / deliverability) — `upsertContact` forces `unsubscribed:false`, can resurrect unsubscribed contacts
**File:** `scripts/lib/broadcast.mjs:26-31` (`upsertContact` → body `{ email, unsubscribed: false }`); every broadcast also creates a **fresh temp audience** (`broadcastToSegment` → `syncAndBroadcast({ audienceId: null })`).
**What's wrong:** Resend honours an audience-level unsubscribe when a recipient clicks its managed unsubscribe link. Because each campaign builds a **new** audience and upserts contacts with `unsubscribed:false` *explicitly*, that prior unsubscribe state is not carried over — the contact is re-activated. The **only** thing preventing a re-send to someone who opted out is the SQL segment's `email_suppressions` exclusion, and a Resend-native broadcast unsubscribe only lands in `email_suppressions` if the new `contact.unsubscribed` webhook handler fires — which requires `RESEND_WEBHOOK_SECRET` set + webhook deployed in prod. **That secret is absent from `.env.local` and unconfirmed in prod (carryover L3 below), and the route now fails *closed* (503) without it** — so today the suppression backstop may not be populating at all.
**Why it matters:** Emailing people who unsubscribed is a CAN-SPAM / GDPR / UK-PECR violation and a direct hit to sender reputation via spam complaints — on a sender whose bounce rate is already **4.57%** (above the risk line). One full re-blast of resurrected unsubscribers could materially raise the complaint rate.
**Suggested fix:** Don't send `unsubscribed:false` on upsert — **omit the field** so Resend preserves the contact's existing opt-out. Prefer a single persistent audience (`RESEND_AUDIENCE_ID`) over per-campaign temp audiences so unsubscribe state accrues in one place. And close the loop: confirm `RESEND_WEBHOOK_SECRET` is set + webhook deployed so `contact.unsubscribed`/bounce events actually reach `email_suppressions`.

## Low

### L1 (NEW — minor) — Temp audiences never cleaned up; mig-55 header doc drift
- `broadcastToSegment` always calls `syncAndBroadcast` with `deleteTempAudience: false`, so every targeted campaign leaves an **orphan audience** in Resend that accumulates over time. Either reuse one persistent audience (preferred, see M1) or set `deleteTempAudience: true` for genuinely one-off sends.
- `migration 55` header comment lists "12 segments" but ships **13** functions — `get_segment_classic_active` is implemented and granted but undocumented in the file header. Pure doc drift; harmless.

### L2 (NEW — informational) — Migration 55 functions are sound, but note `auth.users` PII exposure is service-role-only
All 13 `get_segment_*` functions read `auth.users.email` under `security definer`. The grant block correctly `revoke … from public/anon/authenticated` (handling the PUBLIC-default gotcha) and `grant execute … to service_role`. **No anon/authenticated reach.** The `(p_days || ' days')::interval` pattern is injection-safe because the params are typed `int`. No action needed — recorded so it isn't re-flagged.

### L3 (carryover, still open) — `RESEND_WEBHOOK_SECRET` unconfirmed in prod; webhook now 503s without it
`RESEND_WEBHOOK_SECRET` is **not** in `.env.local` (only `RESEND_API_KEY`, `RESEND_CAMPAIGNS_API_KEY`, `RESEND_AUDIENCE_ID` are). The hardened route returns **503** when it's missing, so if it's also unset in Vercel prod, **all bounce/complaint/unsubscribe suppression silently stops** — and M1 above becomes a live (not hypothetical) re-mail risk. Confirm the secret is set in Vercel and the Resend dashboard shows recent events as *verified*, not 503/401.

### L4 (carryover, still open) — `.env.local.bak.1781514469` still on disk
**Confirmed present** (3033 bytes, Jun 15) in repo root with live secrets. Gitignored now (can't be committed) but should be removed: `rm .env.local.bak.1781514469` and rotate any keys it contains.

## Critical (carryover, still open — top integrity priority)

### C1 — WC quiz answer bank still shipped to the browser
`src/lib/draft/wc-quiz.ts` imports `@/data/draft/wc-quiz.json` and **still lacks `import "server-only"`**; it's imported by `"use client"` pages (`38-0/wc/page.tsx`, `38-0/wc/run/[id]/page.tsx`). The 63-question bank with `answer` indices is in client JS, defeating ranked-board integrity on the £100 prize. Unchanged since Jun 17. Fix: split `POOL`/`deciderQuestion`/`dailyQuestions` into a `server-only` module; add a CI grep forbidding any client import of `wc-quiz.json`.

## High (carryover, still open)

### H-prior — anon key can read quiz answer columns
`supabase/schema.sql` — `challenge_questions`, `questions`, `quiz_packs` are `for select to public using (true)`, so the anon key can `SELECT` answer columns → scripted cheating on the prize board. `scripts/audit-wc2026-integrity.mjs` remains the compensating control. Fix: serve gameplay questions via a server route that omits answers, or split answers into an anon-revoked table.
