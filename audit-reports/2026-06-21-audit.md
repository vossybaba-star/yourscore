## Medium

### M1 (NEW) — Uncommitted WC "catch-up" back-catalog amplifies the open C1 answer-leak, and is unversioned WIP
**Files (working tree, uncommitted):** `src/lib/draft/wc-server.ts` (`pastEditions`, `resolveEdition(catchupDate)`), `src/app/api/draft/wc/route.ts`, `src/app/api/draft/wc/draft/route.ts`, `src/app/38-0/wc/page.tsx`.
**What it is:** A new feature lets a late-joiner play **any** past ranked WC edition (the open back-catalog), not just the immediately-previous one.
**Server-side it is sound:** `catchupDate` is rejected unless it is `< activeEdition` **and** appears in the real `pastEditions` list (derived from actual `draft_wc_runs.run_date`, so no spoofed dates); `lockedFor(user,edition)` still enforces **one go per edition** (no replay farming); routes are authed, `user.id`-scoped, and rate-limited. No IDOR, no injection.
**Why it matters:** It interacts badly with the still-open **C1**. Today the cheat window is bounded to ~2 editions (current + previous); opening the *entire* back-catalog for ranked play means every past edition becomes farmable with the answer bank that is still shipped in client JS — a late entrant could grind the whole catalog toward the £100 board with view-source answers. Also, this code is **uncommitted** in the working tree, so it is currently unversioned and could be lost or shipped half-reviewed.
**Suggested fix:** Do **not** ship the back-catalog catch-up until **C1** is fixed (server-only answer module). Until then, keep ranked catch-up bounded to the previous edition. Commit the WIP to a feature branch so it is versioned and reviewable, not loose in the tree.

**File:** `src/app/api/webhooks/resend/route.ts:15-19` (committed `c7045bd`). The hardened route returns **503** if `RESEND_WEBHOOK_SECRET` is unset — correct for security, but it means bounce/complaint **suppression stops entirely** if the secret is missing in prod. The var is **not in `.env.local`**, and prod-presence is unconfirmed from here.
**Why it matters:** Bounce rate is already **4.57%** (above the risk line per the Jun-12 audit) with 690 permanent bounces that must stay suppressed. If Resend can't deliver signed events (secret missing) the suppression list stops growing and the rate climbs toward a sender-reputation problem. This is the inverse trade-off of fixing the fail-open hole — worth a one-line prod check.
**Suggested fix:** Confirm `RESEND_WEBHOOK_SECRET` is set in Vercel prod, the webhook fix is **deployed**, and the Resend dashboard shows recent events as **verified** (not 503/401). Keep migration 53 (neutered `delete_bounced_user`) as defence-in-depth regardless.
