## Medium

### M-split (NEW) — debate community-split does two unbounded full-table reads on every card render
**Where:** [src/lib/debate.ts:50-59](src/lib/debate.ts) `debateSplit()` runs `select option_idx from debate_votes` **and** `select option_idx from debate_anon_votes` for the debate (no aggregation, no limit) and counts them in JS. It is called by **every** `GET /api/debate/today` ([today/route.ts:24](src/app/api/debate/today/route.ts)) — which is `force-dynamic` + `force-no-store` (uncached) and mounts via `DebateCard` on the **home dashboard, `/versus`, `/debate`, and every `/play/[roomId]` result screen** — and again on **every** `POST /api/debate/vote` ([vote/route.ts:55](src/app/api/debate/vote/route.ts)).
**Why it matters:** anonymous voting exists precisely to drive **viral, account-free volume** (the `?pick=` share funnel is tap → voted). A debate that goes around can accrue tens of thousands of `debate_anon_votes` rows, and **every home-page load then re-scans both entire tables** and ships them to the function to count. It compounds with mild ballot-stuffing: the IP limit is 10/60s but `voter_key` is client-minted, so one IP can insert ~600 fresh anon rows/hour (each a new `(debate_id, voter_key)`), growing the very tables that get full-scanned on every render. (The *integrity* of the split is a documented, accepted trade-off — "banter, not the £100 board"; this finding is about **cost**, not correctness.)
**Fix:** compute the split with an aggregate instead of pulling rows — either a `SECURITY DEFINER` RPC doing `select option_idx, count(*) from (union) group by option_idx` (service-role only, mirrors the existing WC-leaderboard RPC pattern), or add a short `s-maxage` to `/api/debate/today` (the split tolerates a few seconds of staleness; the client already appends the voter's own fresh vote optimistically). Consider a per-debate anon-row cap or bucketing `voter_key` by IP to bound table growth.

### M-comments-update (carryover, UNCHANGED) — `comments` UPDATE RLS policy bypasses the moderation filter
**Where:** [supabase/migrations/70_debates_discussions.sql:63-65](supabase/migrations/70_debates_discussions.sql) — `comments delete own` grants authenticated users full-column UPDATE on their own rows via PostgREST. First-party soft-delete goes through the **service role** ([api/comments/route.ts](src/app/api/comments/route.ts)), so this policy is pure attack surface: post a clean comment (passes `commentRejection`), then `PATCH /rest/v1/comments?id=eq.<own-id>` to rewrite `body` to a spam URL/slur (the door-time filter runs only in POST) or flip `subject_id` onto any thread. Spam/moderation-bypass, **not** XSS (React-escaped). **Fix:** drop `comments delete own` (service role owns the write), or pin `body`/`subject_type`/`subject_id`/`user_id` to old values in a trigger. No change this cycle.

## Provider-limit sanity check
- **New sink:** `debateSplit` full-table reads (M-split) + the still-uncached, now-`force-no-store` `GET /api/comments` (L-comments-get) are the two Supabase-IO amplifiers to watch **before** debate/comment threads see real viral traffic. Both are cheap to cap (aggregate RPC or `s-maxage`). Anon voting removes the sign-up gate, so volume on these endpoints can spike faster than any authed feature has.
- **Net-neutral/positive:** anon voting itself adds one small INSERT per guest vote (IP-limited) — negligible write load; the read side is the concern, not the write side.
- No new hot public GET with a cache-header **regression** beyond the intentional `force-no-store` on live threads; no new realtime channel subscription this cycle (debate/comment components poll via `fetch`, so no effect-cleanup-in-`.then` leak was introduced).
