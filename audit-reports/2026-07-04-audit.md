## Medium

### M1 (NEW) — `/api/auth/validate-email` is unauthenticated with no rate limit → DNS / function amplifier + domain oracle
**Where:** [src/app/api/auth/validate-email/route.ts](src/app/api/auth/validate-email/route.ts), [src/lib/email-verify.ts:51-93](src/lib/email-verify.ts) — Node-runtime route that runs `dns.resolveMx` / `dns.resolve` / `dns.resolve6` on an attacker-controlled domain. No `rateLimitDistributed`, no auth, no domain-length cap.
**Why it matters:** the per-domain `domainCache` only dedups *repeated* domains — a flood of **unique** domains bypasses it entirely. An unauthenticated loop can (a) make the server perform uncached DNS I/O per request, each a billable Vercel Node-function invocation (usage/cost amplifier), (b) point the server's resolver at arbitrary nameservers (server-as-DNS-proxy for recon / light amplification), and (c) use the yes/no response as an existence oracle for any domain. Low blast radius, but it's the one new externally-reachable compute sink this cycle and every other public mutating route here is rate-limited.
**Fix:** wrap in `rateLimitDistributed` keyed by IP (this route has no user yet); cap domain length / reject obviously-malformed before the DNS call; consider a short negative-cache for NXDOMAIN so repeated junk is cheap.

## Low

- **L1 (NEW)** — **`/profile/[userId]` is reachable while logged-out.** [src/app/profile/[userId]/page.tsx:66-67](src/app/profile/[userId]/page.tsx) only redirects when `user?.id === userId`; a `null` (anonymous) session falls straight through to the full render. The header comment says "any **signed-in** player can look up any other player", but nothing enforces sign-in. Consequence: per-user quiz history, recent h2h results and W-D-L become anonymously readable, and the `user_id`→activity mapping is enumerable without an account. **No PII leaks** (the `profiles` select is scoped to `id, display_name, avatar_url`; the `h2h_challenges` select carries no answer columns — both good). **Fix:** if the intent is signed-in-only, add `if (!user) redirect('/login?next=/profile/'+userId)`; otherwise drop the misleading comment and accept it as fully public. Also note the page fires ~6 uncached **service-role** queries per view (RPC + league count + attempts + h2h + packs + profiles) with no cache header — a mild IO amplifier on an anonymously-hammerable URL; a short-TTL cache or `unstable_cache` would help.
- **L2 (NEW)** — **`user_email_segments` view (mig 68) created without `security_invoker=on`** ([supabase/migrations/68_user_email_segments_wc_ranked.sql:13](supabase/migrations/68_user_email_segments_wc_ranked.sql)). It runs with the owner's (RLS-bypassing) privileges — the exact pattern the Jun 18 sprint fixed on `yourscore_user_ratings`. It does **not** project email (only per-profile activity booleans + tiers), so no PII leak, but if the public-schema default grants hand anon/`authenticated` SELECT, those activity flags become anon-readable. **Fix:** `alter view public.user_email_segments set (security_invoker = on);` and confirm no anon grant. Supabase's Security Advisor will also flag this as a WARN.
- **L3 (NEW, governance)** — **`scripts/segments.mjs export-audience` writes plaintext user emails + first names** to `~/Downloads/ys-audiences/*.csv` ([scripts/segments.mjs](scripts/segments.mjs), new `exportAudience`). Not a code vuln, but bulk PII at rest, unencrypted, in a folder that's often cloud-synced. **Fix:** delete the files after upload; keep them out of any Downloads-sync; the code comment already correctly leaves the upload + Customer-List Terms to the founder.
- **L-drift (carryover, grown)** — three applied-but-untracked migrations now: `60_email_sends_log.sql`, **`65_rooms_status_expired.sql`**, **`67_questions_unique_active_text.sql`** (65 + 67 newly applied this cycle; the cleanup-lobbies working-tree edit depends on 65). Commit them so repo state matches prod. Mig high-water is now **68**.
- **L2-webhook-secret (carryover, now higher-stakes)** — `RESEND_WEBHOOK_SECRET` prod presence still unconfirmed. It now gates **more** than bounce suppression: the new `contact.updated` unsubscribe→suppression path also only fires if the webhook is delivered. If the secret is unset the route fail-closes at 503 and **unsubscribes are silently never captured** → re-subscribed on the next audience rebuild. Confirm it's set in Vercel.
