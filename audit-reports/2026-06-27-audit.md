## Executive summary
The entire push-notification stack and the daily-quiz launch pipeline landed on `main` since the last audit (11 commits; the Jun 26 report saw 0). The deployed *server* surface is sound — the cron, the `/api/me/timezone` route, and the `send-push` edge function are all properly authed and fail-closed, and `device_tokens` / `notification_log` correctly deny the anon key. **But one new HIGH was introduced and confirmed live: the temporary `push_debug` diagnostic table is fully readable *and writable* by the anon key** — it leaks user UUIDs + partial device-token tails, and accepts unauthenticated arbitrary inserts. ## High

### H-new — `push_debug` table is anon-readable AND anon-writable in prod
**Where:** table written by [src/lib/push.ts:9-10,32-36](src/lib/push.ts) and [src/components/native/NativeBootstrap.tsx:66-78](src/components/native/NativeBootstrap.tsx). No migration creates it → it was created ad-hoc (SQL editor / Management API) with **no RLS and open grants**.

**Confirmed live this audit** (anon key only):
- `SELECT` returns rows — leaks **full user UUIDs**, the **last 12 chars of real APNs device tokens** (`registration_event` detail), opt-in state, and registration-error strings. 9 rows present at audit time.
- `POST` succeeded — anon inserted an arbitrary row (probe row created and then cleaned up via service role). **Unauthenticated, unbounded writes** to a prod table.

**Why it matters:** (1) Privacy — any visitor can enumerate the UUIDs of users who opened the iOS app plus partial push-token material; (2) Integrity/IO — anyone can flood the table, amplifying Postgres WAL/IO and polluting the very diagnostic data the founder is relying on to fix the device-token bug ([project_yourscore_ios_push]); (3) it is exactly the kind of "temp table left in prod" that the Jun 18 sprint already had to clean up once.

**Fix (do now, even mid-diagnosis):** `alter table push_debug enable row level security;` (deny-all) + `revoke all on push_debug from anon, authenticated;` — the client inserts will then fail, so instead route the `dbg()` writes through a service-role server action, OR accept losing client-side instrumentation. **Once the token issue is confirmed fixed, drop the table and delete the `dbg()` blocks entirely** (tracked in [project_yourscore_ios_push] as cleanup).

### H-prior — anon key can read quiz `answer` columns (fraud path on the £100 board)
**Where:** `questions` (and `quiz_packs`) public `SELECT` policy `USING(true)`.
**Re-confirmed live:** `GET /rest/v1/questions?select=id,answer` with the anon key returns the plaintext `answer` (`"A"`, etc.) for every question. A scripted client can pre-fetch answers for any quiz-gated content, defeating ranked-band/Mastermind integrity. Open since Jun 12.
**Fix:** strip `answer` (and any decider columns) from anon-readable grants — column-level `REVOKE`, or serve questions through a server route that omits the answer; grade server-side only.

## Medium

### M-new — new `profiles.timezone` / `country` location columns are anon-readable
**Where:** migration [57_profile_timezone.sql](supabase/migrations/57_profile_timezone.sql) adds `timezone` + `country` to `profiles`; the existing permissive `profiles` SELECT policy exposes them to the anon key.
**Confirmed:** anon `SELECT id,timezone,country,active_hour_utc` on `profiles` succeeds (values null today because `<TimezoneSync/>` hasn't populated them yet — but it will on next app sessions). At that point **every user's IANA timezone + ISO country becomes anon-enumerable** — coarse location PII keyed to a user id, a privacy regression vs. what `profiles` exposed before.
**Why it matters:** location inference per user, harvestable in bulk with a public key.
**Fix:** column-level `REVOKE` of `timezone, country, active_hour_utc` from `anon` (and `authenticated` for other users), or move the notification-targeting columns to a service-role-only sidecar table. They are only ever read server-side by the cron, so anon/other-user reads are never needed.

### M1 (carryover) — daily push log-before-send can silently drop a whole cohort
[src/lib/notify.ts:54-61](src/lib/notify.ts) still inserts `notification_log` **before** awaiting `send-push`; per-day dedup + single-UTC-hour targeting means any transient `send-push` failure permanently loses that cohort's daily ping with no retry. The code comment acknowledges the trade-off. Unchanged since Jun 24.
**Fix:** log after a confirmed 2xx (accept rare double-send), or delete the just-inserted rows on non-2xx; at minimum surface a `delivered` count.

## Provider-limit sanity
- **New IO/write pressure:** `push_debug` anon-writability (H-new) is a live, unauthenticated write amplifier — small today (9 rows) but trivially floodable. Locking it down removes the risk entirely.
- **Hourly cron** is bounded (≤2000 users/run, one row each in `notification_log`) — negligible.
- **Resend transactional pool** remains the single most likely limit to bite (L1): last over quota at 57,676/50,000, and the daily blast still rides it, directly threatening password-reset / magic-link deliverability. Migrating to Broadcasts is the highest-leverage infra fix.
