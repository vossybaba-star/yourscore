# YourScore — Security & Code Audit · 2026-07-02

**Branch:** `main` @ `d5e2a49` · **Prior report HEAD:** `814511a` · **Reviewer:** automated morning audit (read-only)

## Executive summary
Since yesterday the app got a **Versus rebuild** (rivalry hub, quiz-picker, per-question Quiz-Battle reveal), a batch of **perf routes** (edge-cached `/api/quiz/packs` + `/api/leaderboard/yourscore`, single-call `/api/rank/me`, on-demand 38-0 player pool), a new **`/api/room/invite`** push route, and migrations **62** (h2h answer-detail columns) + **63** (opponent_id indexes). The new server surface is mostly sound — Quiz-Battle grading stays server-authoritative (`h2h/play` still checks each answer against the pack key; `from-attempt` reads the stored attempt), the new edge-cache routes serve only non-PII rows, and no Versus client page writes to the DB directly. **The one genuinely new risk is `/api/room/invite`: it fires a push to any user with no friend check AND no rate limit, and `/api/room/create` is likewise unlimited** — an unbounded push-spam / room-churn path (same class as last week's challenge-create finding, but this instance has *zero* rate limit). Migration 62's new answer columns land in the anon-readable `h2h_challenges` table, but that adds no new answer leak because quiz answers are *already* fully anon-readable (re-confirmed below). The three standing carryovers — `.env.local.bak.*` secrets not gitignored, anon quiz-answer read, and `timezone`/`country` PII — are all **re-confirmed live** this morning; none of the recommended fixes has landed. No code changes made.

### H-answers — anon key reads quiz answers from BOTH `questions` and `quiz_packs` (fraud path, re-confirmed)
**Where:** `questions` public SELECT `using(true)` — [supabase/schema.sql:395](supabase/schema.sql); `quiz_packs` public SELECT `using(true)` — [supabase/schema.sql:426](supabase/schema.sql).
**Re-confirmed live (anon key):** `GET /rest/v1/questions?select=id,answer` returns plaintext answers; **and** `GET /rest/v1/quiz_packs?select=id,questions` returns the full pack with each question's `"answer"` key embedded (`{"answer":"B","options":{…}}`). Scripted clients pre-fetch answers for any quiz-gated / Mastermind / prize-board content. Open since Jun 12.
**Note:** the new `/api/quiz/packs` edge route correctly omits the `questions` column (serves metadata only) — good hygiene — but the fix is cosmetic while the underlying **table RLS still serves questions+answers directly** to anon. The route change does not close the leak; the RLS policy does.
**Fix:** serve pack questions through a server route that strips answers and grades server-side (the game loop already grades server-side), and column-`REVOKE`/policy-scope the raw `answer`/`questions` reads off the anon role.

### H1 — `profiles.timezone` / `country` location PII is anon-readable (carryover, re-confirmed)
**Where:** [supabase/migrations/57_profile_timezone.sql](supabase/migrations/57_profile_timezone.sql) — columns added with no column-level REVOKE, inheriting the permissive public `profiles` SELECT.
**Re-confirmed live (anon key):** `select=username,timezone,country&timezone=not.is.null` returns named coarse-location rows (`{"username":"dreamteam","timezone":"Europe/London","country":"GB"}`). Named, per-user, bulk-enumerable; grows every time `<TimezoneSync/>` runs. Read only server-side (send-window cron).
**Fix:** `REVOKE timezone, country, active_hour_utc ON public.profiles FROM anon, authenticated;` as a one-line migration, or move targeting columns to a service-role-only sidecar table.

## Medium

### M-invite (NEW) — `/api/room/invite` fires push to ANY user with no friend check AND no rate limit
**Where:** [src/app/api/room/invite/route.ts](src/app/api/room/invite/route.ts) — authed, and correctly checks `room.created_by === user.id` (no IDOR on the room), but takes an attacker-chosen `invitedUserId`, does **no** friend/relationship check, and calls `notifyUsers` (opt-in-gated push). **No `rateLimitDistributed`** (confirmed: 0 matches). [src/app/api/room/create/route.ts](src/app/api/room/create/route.ts) is likewise authed with **no rate limit**.
**Abuse path:** loop `POST /api/room/create` (unlimited) → `POST /api/room/invite {roomId, invitedUserId: <stranger>}`. The push `dedupeKey` is `room-invite:${room.id}:${invitedUserId}`, so a **fresh room per call defeats the dedup** — unbounded pushes + "wants to play you live" inbox entries from strangers to any opted-in user. Target `user_id`s are enumerable via the anon-readable leaderboard / `h2h_challenges` / group tables.
**Why it matters:** same notification-abuse / App-Store-review class as last week's challenge-create finding, but this route has *zero* rate limit — the most actionable push finding this week. Opt-in gating caps reach but not abuse. The unlimited `rooms` inserts are also a WAL/IO amplifier (see provider note).
**Fix:** verify a mutual/accepted friend before targeting `invitedUserId` (the `friends` table already backs `/api/friends`); add `rateLimitDistributed` to both `room/invite` and `room/create`; optionally a per-target-per-day notify cap independent of the dedupe key.

### M-answers62 (NEW) — mig 62 adds per-user answer/timing detail to the anon-readable `h2h_challenges`
**Where:** [supabase/migrations/62_h2h_battle.sql](supabase/migrations/62_h2h_battle.sql) adds `challenger_answers`/`opponent_answers` jsonb (`[{letter, correct, elapsedMs}]`) to `h2h_challenges`, whose SELECT policy is `using(true)` ([supabase/schema.sql:393](supabase/schema.sql)).
**Confirmed live (anon key):** both columns are anon-readable (currently `null` — no battle-mode games have completed yet, `count=0`). Once populated they expose, per row and joined to `challenger_id`/`opponent_id`/`quiz_pack_id`, each player's picked letter + correctness + answer timing.
**Why it matters:** the answer-key derivation is **not** a new leak — pack answers are already fully anon-readable via `quiz_packs.questions` (H-answers). What's newly exposed is **per-user answer patterns and response timing tied to `user_id`**, anon-readable and bulk-harvestable — a modest privacy widening. Flagging now, before the columns fill with real data over the coming days.
**Fix:** fold into the `h2h_challenges` policy scoping below (participant-scoped SELECT), or drop the raw answer columns from the anon grant and serve the reveal through a server route by challenge id.

### M-graph (carryover) — `h2h_challenges` + group tables expose a named social graph to anon
[supabase/schema.sql:393](supabase/schema.sql) (`h2h_challenges`) and migration 61's `group_challenge*` tables are all `using(true)`. Anon reads `challenger_name`/`opponent_id`/`display_name`/`score`/`invited` — a bulk "who plays whom" graph and the raw `user_id` set that feeds M-invite's target enumeration. Unchanged from prior reports.
**Fix:** one participant-scoped SELECT policy pattern (readable only when `auth.uid()` is a participant/creator) closes both the h2h and group reads; serve public link-join boards through a server route by id.

### M-notify (carryover) — `notifyUsers` logs-before-send can silently drop a cohort
[src/lib/notify.ts:54-61](src/lib/notify.ts) inserts `notification_log` **before** awaiting `send-push`; a transient failure permanently marks those users "sent" with no retry. Now that room-invite / group / h2h pushes all ride this path, a failed edge call silently eats a batch. Unchanged.
**Fix:** log after a confirmed 2xx (accept rare double-send) or delete the just-inserted rows on non-2xx; at minimum surface a delivered count.

- **Unbounded `rooms` inserts** (M-invite): `/api/room/create` has no rate limit — a churn loop amplifies Postgres WAL/IO and connection pressure, on top of the push-spam. Rate-limiting it addresses both.
