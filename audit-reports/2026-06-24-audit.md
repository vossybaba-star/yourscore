## Medium

### M1 (NEW — correctness / deliverability) — daily push log-before-send silently drops a whole cohort on any `send-push` hiccup
**File:** `src/lib/notify.ts:55-79` — step 3 inserts `notification_log` rows for the whole batch, *then* step 4 fetches the `send-push` edge function. If step 4 returns non-2xx or throws (function cold-start failure, deploy gap, APNs/FCM outage, network), it's logged and swallowed — `targeted` is still returned as success, and the `notification_log` rows are **already committed**.
**Why it matters:** The cron (`/api/cron/wc-mastermind`) targets each user in exactly **one** UTC hour (`active_hour_utc == hour`). The per-day dedup key `wc-mastermind:<date>` means that once the log rows exist, that cohort is **never re-targeted today** — the next hourly run picks a different bucket. So a brief `send-push` failure at, say, 19:00 UTC permanently loses the daily ping for everyone in the 19:00 (and fallback) bucket — the single highest-value notification of the day, on the feature whose entire purpose is a reliable daily drop. The "log before delivery" was chosen for at-most-once safety against cron retries, but the cron isn't retried within the hour, so the tradeoff is all downside here.
**Suggested fix:** Move the `notification_log` insert to **after** a confirmed 2xx from `send-push` (accept the rare duplicate on a true retry — a marketing push tolerates at-least-once far better than silent loss). Or: on a non-2xx delivery, **delete the just-inserted log rows** so the next hourly run (or a manual re-run) can re-target them. At minimum, surface delivery failures (Sentry breadcrumb / return a `delivered` count distinct from `targeted`) instead of swallowing them.

## Low

### L1 (NEW — operational, not a vuln) — push feature is gated on two prod env vars + two unapplied migrations
- The cron no-ops unless **`WC_MASTERMIND_PUSH_ENABLED === "true"`** and 401s unless **`CRON_SECRET`** is set in Vercel prod. Both are correct fail-safe defaults, but mean the feature does nothing until explicitly switched on — worth confirming intentional state before assuming pushes are going out.
- `notify.ts`, the cron, and `/api/me/timezone` all use `supabase as unknown as SupabaseClient` casts because **migrations 56/57 are not yet reflected in generated types** (and may not be applied to prod). If 56/57 aren't applied, every push run fails at the `notification_log`/`active_hour_utc` query and `notifyUsers` returns `{targeted:0}` silently. Apply 56/57, regenerate types, and drop the untyped handles. *(Note: code comments acknowledge this explicitly — flagged so it's tracked to closure, not as a surprise.)*

Not urgent; add a light per-user limit if/when a generic mutating-route limiter exists.
