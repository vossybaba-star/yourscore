## High

### H1 — Prior Critical's protection is uncommitted/undeployed; prod still runs the fail-open webhook
**Files:** `src/app/api/webhooks/resend/route.ts` (working-tree modified, **not committed**); `supabase/migrations/53_neuter_delete_bounced_user.sql`, `51_rls_security_advisor_fixes.sql`, `52_security_advisor_hardening.sql`, `49_get_wc_lapsed_players.sql` (**all untracked**).
**What's wrong:** The durable webhook fix (fail-closed + suppress-only) sits unstaged in the working tree, so **deployed production still serves the old fail-open `route.ts`**. The live mass-deletion hole is currently closed *only* by migration 53 neutering `delete_bounced_user()` in the prod DB — and that migration, plus the RLS/EXECUTE-revoke hardening migrations (51/52) that closed the `increment_profile_score` score-fraud vector and the `admin_chat_messages`/`spend_logs`/`health_logs` RLS holes, are **untracked in git**. A clean migration apply, a DB restore from committed state, or a fresh environment rebuild would silently drop every one of those fixes and leave the fail-open route as the deployed code.
**Why it matters:** The protection against a "delete any account by email" hole and an "anyone grants themselves unlimited prize points" hole is real but fragile — it lives in mutable prod DB state that version control doesn't know about. This is the single highest-leverage item this morning.
**Suggested fix:** (1) Commit the webhook `route.ts` fix and **deploy** it; (2) set `RESEND_WEBHOOK_SECRET` in Vercel prod; (3) commit migrations 49/51/52/53 so the security state is reproducible. After deploy, keep migration 53 as defence-in-depth. (Held back to date because `main` carried other-session WIP — this branch is a clean opportunity to land them.)

### H-prior (carryover, still open) — anon key can read quiz answer columns

## Low

### L1 (carryover) — Username still has no charset/max-length validation
**File:** `src/app/settings/page.tsx` — the Jun-20 settings change was a cosmetic `Button` swap; the username write path is untouched, so only `length < 3` is checked. Username mirrors into `display_name` everywhere (leaderboards, Satori share cards, emails). No XSS (React escapes), but homoglyph/zero-width impersonation and unbounded length remain possible.
**Suggested fix:** Enforce 3–20 chars, `[a-z0-9_]`, normalized lowercase at write time; ideally behind a server route so PostgREST can't be hit directly.

### L2 — `cleanup-lobbies` cron depends on `CRON_SECRET` being set in Vercel
**File:** `src/app/api/cron/cleanup-lobbies/route.ts:25`. The guard correctly **fails closed** (401 if `CRON_SECRET` unset) — good. But `CRON_SECRET` is **not in `.env.local`** (local runs will 401), and if it is ever unset in Vercel prod the sweep silently stops running (availability, not security). The existing `comeback`/`reclassify` crons share the same guard and presumably run, so the secret is likely present in prod — worth a one-line confirmation. No vulnerability; noted for completeness.

### L3 (carryover) — Bulk email volume vs Resend cap
Daily WC sends + the lapsed-player campaign keep adding volume against the 50k/mo plan (renews Jul 8). Now that the webhook is suppress-only, a bounce spike no longer mass-deletes accounts — but watch the monthly count and bounce rate. Segment to engaged users if usage climbs.
