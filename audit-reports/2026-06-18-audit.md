## Executive summary
One **new Critical**: the Resend webhook (`/api/webhooks/resend`) **fails open** — when `RESEND_WEBHOOK_SECRET` is unset it processes *unsigned* events, so an unauthenticated POST of `{type:"email.bounced", data:{to:["victim@…"]}}` permanently deletes that user's whole account by email (no UUID needed). The secret is **not** in `.env.local`; its presence in Vercel prod must be confirmed today. Even with the secret set, the design is one typo away from an open mass-deletion API, and it currently hard-deletes on **any** bounce type — a transient "mailbox full" bounce destroys a real user. The new **comments** layer and **username** flow are otherwise sound (authed, rate-limited, server-authoritative, no XSS — React escapes; no `dangerouslySetInnerHTML`). All of yesterday's open items (**C1** bundled answer bank, **H1** `.env.local.bak`, **M1/M2/M3**) remain unfixed.

## Critical

### C-new — Resend webhook fails open → unauthenticated, remote account deletion by email
**File:** `src/app/api/webhooks/resend/route.ts:12-13,35-36,57-61`
**What's wrong:** Signature verification is gated behind `if (secret) { … }`. If `RESEND_WEBHOOK_SECRET` is unset, control falls straight to `const event = await req.json(); return handleEvent(event)` — **no signature check**. `handleEvent` then, on `email.bounced`/`email.complained`, resolves the account from the email in the payload and calls `delete_bounced_user(email)` → `delete_user_account(uuid)` (wipes all public-schema data) → `auth.admin.deleteUser(uuid)` (removes the auth identity). The secret is **not present in `.env.local`** (confirmed); whether it is set in Vercel production is unverified.
**Why it matters:** If the secret is absent in prod, anyone can POST `{"type":"email.bounced","data":{"to":["target@gmail.com"]}}` to `https://yourscore.app/api/webhooks/resend` and **permanently delete any user's account**. No auth, no UUID — just an email address, which is harvestable/guessable. A short script could wipe the entire known user base. Even if the secret *is* configured, fail-open means one env-var deletion/typo silently turns this into an open deletion endpoint.
**Suggested fix:** **Fail closed.** If `RESEND_WEBHOOK_SECRET` is missing, reject every request (`503`/`401`) and never call `handleEvent` on an unverified body. Make verification mandatory in all environments. Then verify the secret is actually set in Vercel prod **today**; if the webhook has been live and unsigned, check Resend/Supabase logs for any unexpected `delete_user_account` calls. (See also H-new below — deletion should not be this eager even with valid signatures.)

## High

### H-new — Account is hard-deleted on ANY bounce type (incl. transient/soft bounces)
**File:** `src/app/api/webhooks/resend/route.ts:42-62`
**What's wrong:** `handleEvent` deletes on `email.bounced` without inspecting the bounce class. It captures `data.bounce_type` into `detail` but never gates on it — a **transient/soft** bounce (full mailbox, greylisting, temporary DNS failure) triggers the same irreversible `delete_user_account` + `auth.admin.deleteUser` as a permanent hard bounce.
**Why it matters:** Legitimate, paying-attention users get their entire account + game history destroyed because their inbox was temporarily full. Combined with the daily WC blast (high send volume), false-positive deletions are likely, not theoretical — and they're unrecoverable.
**Suggested fix:** Only hard-delete on confirmed permanent failures (Resend `bounce_type`/SES `Permanent`/`HardBounce`). For soft bounces, suppress and/or count toward a threshold before any deletion. For `email.complained`, prefer suppress-only or a soft-delete grace window over immediate hard deletion (see M-new).

### M-new — `email.complained` deletes the entire account (aggressive, irreversible data loss)
**File:** `src/app/api/webhooks/resend/route.ts:42-62`
**What's wrong / why it matters:** A single spam complaint nukes the user's account, game history, league memberships, and auth identity. Suppression already stops future email; full deletion is disproportionate and unrecoverable, and users who hit "spam" rarely intend to delete their account and all stats.
**Suggested fix:** On complaint, suppress (already done) and stop there, or apply a reversible soft-delete with a grace period. Reserve hard deletion for explicit user request or confirmed permanent hard bounces.

## Low

### L-new — Username has no charset/max-length validation
**Files:** `src/app/settings/page.tsx:85-92` (only `length < 3` checked); `src/components/profile/UsernamePrompt.tsx:65-72` (min 3, no max, no charset).
**What's wrong / why it matters:** Username is now the public identity and is mirrored into `display_name`, which renders everywhere — leaderboards, league boards, Satori share cards, emails. With no charset allowlist or max length, a user can set homoglyph/zero-width-character handles to impersonate others, or an arbitrarily long string that bloats the row and breaks card/email layout. (No XSS — React escapes — but impersonation and layout abuse are real.) The availability check also passes raw input to `ilike`, so `%`/`_` act as wildcards (cosmetic only).
**Suggested fix:** Enforce a server-checkable rule (e.g. 3–20 chars, `[a-z0-9_]`, normalized lowercase) at write time; reject control/zero-width chars. Ideally move the write behind a small server route so validation + uniqueness can't be bypassed by hitting PostgREST directly.

### L-new — Email local-part can become the public display name
**File:** `supabase/migrations/46_profile_display_name_default.sql:19-21`
**What's wrong / why it matters:** When no OAuth name is present, the default display name falls back to `split_part(email,'@',1)` — e.g. `john.doe` from `john.doe@gmail.com` becomes the public handle until the user picks one. Minor PII exposure; the username prompt mitigates by nudging a choice.
**Suggested fix:** Default to a neutral handle (e.g. `player_<short-id>`) instead of the email local-part.
