## Medium

### M1 (NEW) — `comments` UPDATE RLS policy lets authenticated users bypass the comment moderation filter
**Where:** [supabase/migrations/70_debates_discussions.sql:63-65](supabase/migrations/70_debates_discussions.sql) — `create policy "comments delete own" ... for update using (auth.uid() = user_id) with check (auth.uid() = user_id)`. This grants an authenticated user full-column UPDATE on their own comment rows via the anon/authenticated PostgREST endpoint.
**Why it matters:** the app's own soft-delete goes through the **service role** ([api/comments/route.ts:100-104](src/app/api/comments/route.ts)), so this UPDATE policy is never used by first-party code — it's pure attack surface. The door-time moderation (`commentRejection`: blocks links + slurs, [src/lib/moderation.ts](src/lib/moderation.ts)) runs **only in the POST handler**. A user can post a clean comment (passes the filter), then `PATCH /rest/v1/comments?id=eq.<own-id>` to rewrite `body` to a spam URL or slur — the DB `char_length 1..280` CHECK still applies, but the filter does not. They can also flip `subject_id` to move a comment onto any pack/debate thread. **No XSS** (the thread renders `{c.body}` as a text child — React-escaped, [DiscussionThread.tsx:149](src/components/debate/DiscussionThread.tsx)). Blast radius is spam/moderation-bypass + comment-thread integrity, not code execution.
**Fix:** the policy only needs to permit soft-delete, and even that now runs service-side — so **drop `comments delete own`** entirely (service role already owns the write), or scope it so only `deleted_at` may change (e.g. a trigger/`with check` that pins `body`, `subject_type`, `subject_id`, `user_id` to their old values). Re-run the door filter on any future edit path.

_(Carryover Mediums **M-validate-email / M-invite / M-graph / M-answers62 / M-notify** unchanged — see "Still-open". The `comments`/`debate_votes` `user_id` columns add a little more `user_id`→activity enumeration surface, mildly amplifying M-graph.)_
