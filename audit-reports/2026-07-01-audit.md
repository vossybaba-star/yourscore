## Executive summary
Two feature streams landed since yesterday: the **Versus tab reshape** (Phases A–C: image feed, in-place Friends/Leagues, quiz-first challenge, invite links) and **Async Multiplayer Phase 2 — group challenges** (migration 61, four new `/api/challenge/*` routes, `/g/[id]` board). **The new server surface is again sound in the ways that matter:** every write goes through the service role, group play is graded server-authoritatively against the pack answers (mirrors `/api/h2h/play`), routes are authed + rate-limited, and the expire cron fails closed without `CRON_SECRET`. The genuinely new risk this week is a **push-spam / inbox-injection vector**: all three challenge-create routes accept an arbitrary `invitedUserId` with **no friend/relationship check** and now fire an opt-in push to the target — and `/api/challenge/create` amplifies it to 50 invitees per call. It chains with a **new anon-readable social graph** (migration 61's group tables are `USING(true)`), which hands an attacker every `user_id` to target. Separately, the recurring **`.env.local.bak.*` secret-leak HIGH is worse today** — a *second* backup file appeared Jun 29 and neither is gitignored (the "gitignore hardened" claim in memory never actually landed). The two standing HIGHs (anon quiz-`answer` read; `timezone`/`country` PII) are re-confirmed live. No code changes made.

## High

### H-bak — TWO `.env.local.bak.*` files with live secrets are NOT gitignored (regressed — a second one appeared)
**Where:** repo root — `.env.local.bak.1781514469` **and new** `.env.local.bak.1782746689` (created Jun 29 16:24). Each contains **12** secret-bearing lines (service-role key, webhook/API secrets).
**Confirmed this audit:** `git check-ignore` returns nothing for both → **not ignored**. `.gitignore:29` is still `.env*.local`, which matches `.env.local` but **not** `.env.local.bak.<n>` (the suffix falls outside the glob). The prior-audit note that gitignore was "hardened to `.env*`+`*.bak*`" is **not reflected in the actual file** — that fix never landed. A single `git add -A && commit && push` puts live service-role keys into the public GitHub repo (`vossybaba-star/yourscore`).
**Why it matters:** service-role key = full DB bypass of every RLS policy in this report. This is the highest-actionability item today, and the fact that a *new* `.bak` keeps appearing means whatever process writes them is still running.
**Fix (2 min):** `rm .env.local.bak.*`, then add `.env*` and `*.bak*` to `.gitignore` (verify with `git check-ignore`). Find and stop whatever writes the timestamped `.bak` (looks like an env-editing script). If either was ever committed, rotate the service-role key.

### H-prior — anon key can read quiz `answer` columns (fraud path on the £100 board)
**Where:** `questions` (and `quiz_packs`) public SELECT policy `USING(true)` — [supabase/schema.sql:395](supabase/schema.sql).
**Re-confirmed live this audit (anon key):** `GET /rest/v1/questions?select=id,answer` returns plaintext answers (`"A"`,`"B"`,…) for every question. Scripted clients can pre-fetch answers for any quiz-gated content, defeating ranked-band / Mastermind integrity. Open since Jun 12.
**Note:** the new group-challenge flow is *not* made worse — `/api/challenge/play` re-grades server-side from `quiz_packs.questions` via the service role, so knowing answers still can't forge an opponent's stored score. The exposure remains the prize-board ranked content.
**Fix:** column-level `REVOKE` on `answer`/decider columns, or serve questions through a server route that omits the answer and grades server-side only.

### H1 — `profiles.timezone` / `country` location PII is anon-readable (carryover, still live)
**Where:** [supabase/migrations/57_profile_timezone.sql](supabase/migrations/57_profile_timezone.sql) — columns added with no column-level REVOKE, inheriting the permissive public `profiles` SELECT.
**Re-confirmed live this audit:** `select=username,timezone,country&timezone=not.is.null` returns named coarse-location rows with the anon key (e.g. `{"username":"dreamteam","timezone":"Europe/London","country":"GB"}`). Named, per-user, bulk-enumerable; grows every time `<TimezoneSync/>` runs. Read only server-side (the send-window cron).
**Fix:** `REVOKE timezone, country, active_hour_utc ON public.profiles FROM anon, authenticated;` (as a one-line migration so grant state lives in the repo), or move targeting columns to a service-role-only sidecar table.

## Medium

### M-new1 — push-spam / inbox-injection: challenge-create routes target ANY user with no friend check (now wired to live push)
**Where:** all three creation routes take an attacker-chosen target and fire `notifyUsers`:
- [src/app/api/challenge/create/route.ts](src/app/api/challenge/create/route.ts) — `invitedUserIds` (up to **50** per call, rate limit **15/60s**) → 50 injected `group_challenge_participants` rows + 50 pushes per call.
- [src/app/api/h2h/from-attempt/route.ts](src/app/api/h2h/from-attempt/route.ts) and [src/app/api/h2h/create/route.ts](src/app/api/h2h/create/route.ts) — single `invitedUserId`, rate limit 20/60s each.

None of them checks that the target is a friend (or has any relationship to the caller); the only guard is `invitedUserId !== self`. Each new challenge uses a per-challenge `dedupeKey`, so the notify-log dedup does **not** stop repeat sends — an attacker just creates fresh challenges. Ceiling via `/api/challenge/create`: **~750 pushes + 750 inbox rows per minute** aimed at arbitrary opted-in users, plus stranger names appearing in victims' Your-Turns inbox.
**Why it matters:** this was latent when flagged as h2h "L-new" (email was a TODO). It's now **active** — push is wired in all three routes. It's a harassment / notification-abuse vector and an App-Store-review risk (unsolicited push from strangers). It **chains with M-new2**: the anon social-graph read below hands an attacker every `user_id` to feed as targets. The opt-in gate limits reach but doesn't remove the abuse.
**Fix:** before targeting, verify a mutual/accepted friend relationship for each `invitedUserId` (the friends table already exists — `/api/friends` uses it). Reject non-friends server-side. Optionally add a per-target-per-day notify cap independent of the per-challenge dedupe key.

### M-new2 — group-challenge tables expose a named social graph + scores to the anon key (new, migration 61)
**Where:** [supabase/migrations/61_group_challenges.sql](supabase/migrations/61_group_challenges.sql) — both `group_challenges` and `group_challenge_participants` are `for select to public using (true)`.
**Confirmed live (anon key), real production rows:**
`GET /rest/v1/group_challenge_participants?select=challenge_id,user_id,display_name,score,invited` returns e.g. `{"user_id":"35d1…","display_name":"cuddy","score":2625,"invited":false}` for every participant, joinable by `challenge_id` and to `group_challenges.creator_id`/`creator_name`. This is the same regression class as last week's `h2h_challenges` M-new: a bulk-harvestable "who's playing with whom", per-user scores, and the raw `user_id` set. `invited:true` rows additionally reveal who was *directly* invited.
**Why it matters:** no email/phone/location here (lower than H1), but it's a fresh privacy widening and the `user_id` enumeration is exactly what M-new1 needs to target strangers.
**Fix:** scope both SELECT policies to participants + open/link discovery, e.g. participants readable only when `auth.uid()` is the creator or a participant of that `challenge_id`; serve the public link-join board through a server route (service role, by id) rather than a blanket anon read.

### M1 (carryover) — `notifyUsers` logs-before-send can silently drop a cohort (now covers challenge pushes)
[src/lib/notify.ts:54-61](src/lib/notify.ts) inserts `notification_log` **before** awaiting `send-push`; a transient send failure permanently marks those users "sent" with no retry and no `delivered` count. Now that group/h2h invites ride this path, a failed edge-function call silently eats a batch of challenge pings. Unchanged from prior reports.
**Fix:** log after a confirmed 2xx (accept rare double-send), or delete the just-inserted rows on non-2xx; at minimum surface a delivered count.
